Purpose: These three changes target ~27-45% of pipeline overhead from unnecessary memcpy operations that Redis does not perform.
Output: Modified connection.rs with zero-copy write path, optimized read path, and pre-allocated frames vector.

   Strategy: Replace the BytesMut read_buf with a Vec<u8> read_vec that serves as both the monoio I/O buffer AND the parse source. Since codec.decode_frame expects &mut BytesMut, wrap the Vec content into BytesMut only when needed for parsing, OR better: read into the Vec, then use a single extend_from_slice but eliminate the tmp_buf entirely by reading directly into a slice of read_buf.

   Simplest correct approach — eliminate tmp_buf, read directly into read_buf's spare capacity:
   a. Remove `let mut tmp_buf = vec![0u8; 8192];` (line ~3150)
   b. Before the read call, ensure read_buf has spare capacity: `read_buf.reserve(8192);`
   c. Create a temporary Vec from the spare capacity for the read:
      ```rust
      let spare_len = read_buf.capacity() - read_buf.len();
      let mut read_vec = Vec::with_capacity(spare_len);
      // SAFETY: monoio will write into this buffer and we'll set_len after
      unsafe { read_vec.set_len(spare_len); }
      let (result, returned_vec) = stream.read(read_vec).await;
      match result {
          Ok(0) => break,
          Ok(n) => {
              read_buf.extend_from_slice(&returned_vec[..n]);
          }
          Err(_) => break,
      }
      ```

   Actually, this still has one copy. The truly zero-copy approach: keep read_buf as BytesMut for codec compatibility, but eliminate the SEPARATE tmp_buf allocation. Use a single reusable Vec<u8> that we read into and copy from (which is what we have, BUT fix the resize zero-fill):

   REVISED simpler approach — just fix the zero-fill waste:
   a. Change `tmp_buf.resize(8192, 0);` to:
      ```rust
      // SAFETY: monoio will overwrite the buffer contents; no need to zero-fill.
      // The buffer was allocated with vec![0u8; 8192] initially and we only read
      // up to its capacity, so the allocation is valid.
      if tmp_buf.len() < 8192 {
          tmp_buf.reserve(8192 - tmp_buf.len());
          unsafe { tmp_buf.set_len(8192); }
      }
      ```
   This eliminates the zero-fill memset on every read cycle (~8KB zeroing per read).
