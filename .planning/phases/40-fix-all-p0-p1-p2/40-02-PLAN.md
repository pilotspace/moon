  <action>
Two targeted changes:

**1. Reuse read buffer in Monoio handler (src/server/connection.rs):**

Move the `vec![0u8; 8192]` allocation outside the main loop. The buffer must be declared before the loop and reused each iteration. Monoio's ownership I/O takes ownership of the buffer and returns it, so we reassign.

Find the main loop in handle_connection_sharded_monoio (around line 3299 `loop {`). Before the loop starts, add:
```rust
let mut tmp_buf = vec![0u8; 8192];
```

Then inside the loop, replace line 3350:
```rust
// OLD: let tmp_buf = vec![0u8; 8192];
// NEW: reuse buffer, reset length for next read
tmp_buf.resize(8192, 0);
```

The existing line `let (result, tmp_buf) = stream.read(tmp_buf).await;` needs to reassign back:
```rust
let (result, returned_buf) = stream.read(tmp_buf).await;
tmp_buf = returned_buf;
```

This changes `tmp_buf` from immutable (let) to mutable reassignment. The key insight: monoio's `read()` takes ownership of the Vec and returns it (possibly with different len). We just need to keep the returned buffer for the next iteration instead of dropping it.

to:

This matches CHANNEL_BUFFER_SIZE (line 13) and supports 5K+ concurrent clients without overflow.

  <action>
Add a per-connection pool of oneshot channels to avoid allocating a new oneshot pair on every remote batch dispatch. The pool stores previously-used OneshotSender/OneshotReceiver pairs that can be recycled.

**Implementation approach:**

Since `channel::oneshot()` creates a new allocation each time, and the Monoio handler may dispatch to multiple shards per batch, create a simple pool:

1. Before the main loop, declare a channel pool:
```rust
// Pool of reusable oneshot channels for remote batch dispatch.
// After each batch, receivers are consumed (awaited) and senders are dropped.
// Since oneshot channels are consumed on use, pooling them requires creating
// new ones each time. Instead, optimize by pre-allocating the Vec containers
// that hold reply_futures to avoid per-batch Vec allocation.
let mut reply_futures_buf: Vec<(
    Vec<(usize, Option<Bytes>, Vec<u8>)>,
    channel::OneshotReceiver<Vec<Frame>>,
)> = Vec::with_capacity(num_shards);
```

2. In the batch dispatch section (added in Plan 01), instead of creating a new `reply_futures` Vec each batch:
```rust
// Reuse pre-allocated buffer
reply_futures_buf.clear();
// ... push to reply_futures_buf instead of reply_futures
```

3. Also pre-allocate the remote_groups HashMap capacity outside the loop:
```rust
let mut remote_groups: std::collections::HashMap<usize, Vec<(usize, std::sync::Arc<Frame>, Option<Bytes>, Vec<u8>)>> = std::collections::HashMap::with_capacity(num_shards);
```
And inside the loop, use `remote_groups.clear()` instead of creating a new HashMap.

4. Similarly, pre-allocate the responses Vec outside the loop:
```rust
let mut responses: Vec<Frame> = Vec::with_capacity(64);
```
And inside the loop, use `responses.clear()` + extend/push instead of creating new.

This eliminates per-batch Vec allocations for reply_futures, remote_groups, and responses. The oneshot channels themselves cannot be pooled (they are single-use by design), but the container allocations are the main overhead.

Run `cargo test` to verify no regressions.
