**2. Add pub/sub subscriber mode loop** at the top of the main `loop`, BEFORE the read:
When `subscription_count > 0`, enter a `monoio::select!` loop mirroring the tokio subscriber loop (lines 149-323). The monoio version must use monoio's ownership-based I/O instead of Framed:

```rust
if subscription_count > 0 {
    let rx = pubsub_rx.as_mut().unwrap();
    // Need to read from stream AND receive pubsub messages simultaneously
    let tmp_buf = vec![0u8; 8192];
    monoio::select! {
        read_result = stream.read(tmp_buf) => {
            let (result, buf) = read_result;
            // parse frames from buf, handle SUBSCRIBE/UNSUBSCRIBE/PSUBSCRIBE/PUNSUBSCRIBE/PING/QUIT/HELLO
            // Mirror the exact command handling from tokio subscriber loop (lines 155-296)
            // Use codec.decode_frame() + codec.encode_frame() instead of framed.next()/framed.send()
            // Write responses via stream.write_all()
        }
        msg = rx.recv_async() => {
            // Received published message, encode and send to client
            match msg {
                Ok(frame) => {
                    let mut resp_buf = BytesMut::new();
                    codec.encode_frame(&frame, &mut resp_buf);
                    let data = resp_buf.to_vec();
                    let (result, _) = stream.write_all(data).await;
                    if result.is_err() { break; }
                }
                Err(_) => break,
            }
        }
        _ = shutdown.cancelled() => { break; }
    }
    continue;
}
```

NOTE: monoio::select! works like tokio::select! for futures. The `stream.read(tmp_buf)` returns `(Result<usize>, Vec<u8>)` — the ownership model. Handle the `(result, buf)` tuple properly.

**3. Add SUBSCRIBE/PSUBSCRIBE handling** in the normal command processing section (inside the `for frame in &frames` loop):
When SUBSCRIBE/PSUBSCRIBE is received outside subscriber mode:
- Allocate pubsub channel if not yet created: `let (tx, rx) = channel::mpsc_bounded(256); pubsub_tx = Some(tx); pubsub_rx = Some(rx);`
- Allocate subscriber_id if zero: `subscriber_id = crate::pubsub::next_subscriber_id();`
- Register subscriber, increment subscription_count, send confirmation response
- After processing the SUBSCRIBE frame, flush any accumulated responses and `continue` the outer loop (so next iteration enters subscriber mode)

**4. Add MULTI/EXEC/DISCARD intercepts** in the normal command processing loop, BEFORE the routing/dispatch section:
- MULTI: set `in_multi = true`, push OK response

- DISCARD: set `in_multi = false`, clear queue, push OK
- When `in_multi` and command is not EXEC/DISCARD/MULTI: push frame to `command_queue`, respond QUEUED
