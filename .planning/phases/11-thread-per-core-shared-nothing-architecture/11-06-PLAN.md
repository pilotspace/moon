1. Add a `SnapshotRequest` variant to `ShardMessage` in `src/shard/dispatch.rs`:
```rust
/// Request this shard to clone its database state for RDB snapshot.
SnapshotRequest {
    reply_tx: tokio::sync::oneshot::Sender<Vec<(Vec<(Bytes, Entry)>, u32)>>,
},
```

The reply contains per-database data: `Vec<(entries, base_ts)>` for each of the 16 databases in this shard.

2. Handle `SnapshotRequest` in the shard event loop (`src/shard/mod.rs`):
```rust
ShardMessage::SnapshotRequest { reply_tx } => {
    let snapshot: Vec<(Vec<(Bytes, Entry)>, u32)> = self.databases.iter().map(|db| {
        let base_ts = db.base_timestamp();
        let entries: Vec<(Bytes, Entry)> = db.data().iter()
            .map(|(k, v)| (k.clone(), v.clone()))
            .collect();
        (entries, base_ts)
    }).collect();
    let _ = reply_tx.send(snapshot);
}
```

3. BGSAVE command handling: When BGSAVE is triggered (at the connection level), the coordinator:
   a. Sends `SnapshotRequest` to all shards (including the local one via direct access)
   b. Collects all responses
   c. Merges per-shard snapshots into a single dataset
   d. Calls `rdb::save_from_snapshot()` on a background task

For Phase 11, keep BGSAVE simple: it requests snapshots sequentially from each shard. This means the snapshot is NOT perfectly point-in-time across shards (keys modified between shard snapshots are inconsistent). This is acceptable -- Phase 14 implements true consistent snapshots. Add a comment noting this.

4. RDB load on startup: Currently `rdb::load()` populates `&mut [Database]`. For sharded startup:
   - Load RDB into temporary databases (same as current code)
   - For each key in the loaded databases, compute `key_to_shard(key, num_shards)`
   - Route key to the correct shard's databases
   - This happens during bootstrap in `main.rs`, BEFORE shard threads start, so no cross-shard dispatch needed.

Add a helper function:
```rust
pub fn distribute_loaded_to_shards(
    loaded_dbs: Vec<Database>,
    num_shards: usize,
    shard_dbs: &mut [Vec<Database>],  // shard_dbs[shard_id][db_index]
) {
    for (db_idx, db) in loaded_dbs.into_iter().enumerate() {
        for (key, entry) in db.data().iter() {
            let target_shard = key_to_shard(key, num_shards);
            shard_dbs[target_shard][db_idx].set(key.clone(), entry.clone());
        }
    }
}
```

  <action>
AOF design for Phase 11: **Single global AOF writer** receiving entries from all shards.

1. The AOF writer task runs on the listener thread (or a dedicated background thread). It receives `AofMessage::Append` from all shards via a single tokio mpsc channel.

2. Each shard gets a clone of the `mpsc::Sender<AofMessage>` (this is the one case where we share something across threads -- but `mpsc::Sender` is designed to be shared).

3. In the sharded connection handler, AOF logging works the same way as before:
   - Serialize write commands before dispatch
   - After successful execution (no error response), send to `aof_tx`
   - The `aof_tx` is passed to each shard thread during bootstrap

4. For `BGREWRITEAOF`: Same pattern as BGSAVE -- request snapshots from all shards, merge, generate synthetic commands, write to file.

5. AOF startup replay: Same as RDB load -- replay into temporary databases, then distribute keys to shards.

No structural changes to `aof.rs` needed -- the `AofMessage` enum and `aof_writer_task` work as-is. The mpsc channel naturally handles multiple producers (one per shard thread).

The main change is in the bootstrap code (main.rs) and the sharded connection handler -- passing `aof_tx: Option<mpsc::Sender<AofMessage>>` to each shard, and each shard passes it to connection handlers.
