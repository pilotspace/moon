Add bumpalo arena to `handle_connection()` for per-batch temporary allocation. The arena owns a single allocation pool that grows as needed and is bulk-reset in O(1) after each batch cycle.

**Important: Arena scope clarified per research findings.** CONTEXT.md originally stated "Arena used for: parsed command argument vectors, temporary string slices during inline parsing." Research (Pattern 5, Anti-Patterns) confirmed that `Frame::Array(Vec<Frame>)` cannot use arena allocation because Frame must outlive the parse phase -- changing Frame's Vec type is forbidden (API contract). Similarly, the `responses`, `dispatchable`, and `aof_entries` batch vectors contain owned types (`Frame`, `Bytes`) that are sent downstream and must outlive the arena reset. The arena is therefore used as:

1. **Infrastructure** -- established per-connection with O(1) reset after each batch, ready for Phase 9+ per-request temporaries.
2. **Concrete scratch allocation** -- use `bumpalo::collections::Vec` for a small scratch buffer during inline token assembly in the batch loop, demonstrating the arena pattern. Specifically, allocate a `bumpalo::collections::Vec<&[u8]>` to collect token slices before converting them to owned `Frame::BulkString` values.

**1. Add imports at top of connection.rs:**

**2. Create arena in `handle_connection()`, after the existing variable declarations (after line ~96 where `watched_keys` is declared):**

```rust
// Per-connection arena for batch processing temporaries.
```

**3. Add `arena.reset()` call after each batch cycle completes:**

The arena.reset() should be placed just before the closing brace of the `first_result = framed.next()` arm of the `tokio::select!`, right before the `if break_outer || should_quit { break; }` check. Specifically, insert it after the AOF entries loop (line ~688) and before the break check (line ~690).

**Note on std::vec::Vec usage (intentional per research):** The `responses`, `dispatchable`, and `aof_entries` vectors remain `std::vec::Vec` because:
- `responses` elements are `Frame` which must outlive the batch (they're sent via framed.send)
- `dispatchable` elements contain `Frame` and `Option<Bytes>` that must outlive parse
- `aof_entries` elements are sent to the AOF channel
- Research Pattern 5 and Anti-Patterns explicitly confirm: "Frame::Array(Vec<Frame>) uses std::vec::Vec... Arena-allocated Vecs cannot escape into Frame"

This is NOT a contradiction with the CONTEXT.md arena decision -- see updated CONTEXT.md which clarifies the arena serves as infrastructure in Phase 8 with concrete scratch usage where lifetime permits.

```rust
fn bench_parse_pipeline_32(c: &mut Criterion) {
    let config = ParseConfig::default();
    // Build a 32-command pipeline: *3\r\n$3\r\nSET\r\n$5\r\nmykey\r\n$7\r\nmyvalue\r\n x32
    let single_cmd = b"*3\r\n$3\r\nSET\r\n$5\r\nmykey\r\n$7\r\nmyvalue\r\n";
    let mut pipeline_data = Vec::with_capacity(single_cmd.len() * 32);
    for _ in 0..32 {
        pipeline_data.extend_from_slice(single_cmd);
    }

    c.bench_function("parse_pipeline_32cmd", |b| {
        b.iter(|| {
            let mut buf = BytesMut::from(&pipeline_data[..]);
            let mut count = 0;
            while let Ok(Some(frame)) = parse(black_box(&mut buf), &config) {
                black_box(&frame);
                count += 1;
            }
            assert_eq!(count, 32);
        })
    });
}
```
