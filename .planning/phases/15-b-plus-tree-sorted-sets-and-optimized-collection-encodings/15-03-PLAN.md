**Strategy for accessors:** The existing accessor methods (get_or_create_hash, get_or_create_list, etc.) return `&mut HashMap`/`&mut VecDeque`/etc. Command handlers depend on this signature. To minimize disruption:

1. When a NEW key is created, start with compact encoding (listpack/intset)
2. When a MUTABLE accessor is called on a compact-encoded key, upgrade to full encoding first, then return the full-size reference as before
3. When a READ-ONLY accessor is called on a compact-encoded key, upgrade to full encoding too (for simplicity -- command handlers expect full types)
4. This means command handlers DO NOT need changes for hash/list/set -- they always see the full type

For sorted sets specifically: the return type changes from `(&mut HashMap, &mut BTreeMap)` to `(&mut HashMap, &mut BPTree)`. This is a breaking change that Plan 04 addresses in sorted_set.rs.

**Update `get_or_create_hash`:**
```rust
pub fn get_or_create_hash(&mut self, key: &[u8]) -> Result<&mut HashMap<Bytes, Bytes>, Frame> {
    // ... expiry check (unchanged) ...
    if !self.data.contains_key(key) {
        // NEW: start as listpack
        let entry = Entry::new_hash_listpack();
        let k = Bytes::copy_from_slice(key);
        self.used_memory += entry_overhead(key, &entry);
        self.data.insert(k, entry);
    }
    let entry = self.data.get_mut(key).unwrap();
    match entry.value.as_redis_value_mut() {
        Some(RedisValue::Hash(map)) => Ok(map),
        Some(RedisValue::HashListpack(lp)) => {
            // Upgrade: convert listpack to HashMap
            let map = lp.to_hash_map();
            *entry.value.as_redis_value_mut().unwrap() = RedisValue::Hash(map);
            match entry.value.as_redis_value_mut() {
                Some(RedisValue::Hash(map)) => Ok(map),
                _ => unreachable!(),
            }
        }
        _ => Err(Self::wrongtype_error()),
    }
}
```

**Update read-only _if_alive accessors** (get_hash_if_alive, etc.):
These return immutable references. For compact variants, we need to upgrade since we can't return `&HashMap` from a Listpack. However these take `&self` not `&mut self`. Two options:
- Change to take `&mut self` (breaks API)
- Return a new enum that can be either type

For simplicity and to avoid massive API changes: make the read-only _if_alive methods also return WRONGTYPE error for compact variants -- they should have been upgraded before read-path access. In practice, the mutable path always runs first (creating as listpack, then upgrading on first mutable access). If a read-only path encounters a compact encoding, it means the key was just created and never mutated -- in that case the listpack is empty or very small and upgrading is cheap. So: add a note that compact encodings in read-only path should not occur in normal flow, but handle them gracefully by... actually the simplest approach is to match on both variants:

```rust
pub fn get_hash_if_alive(&self, key: &[u8], now_ms: u64) -> Result<Option<&HashMap<Bytes, Bytes>>, Frame> {
    // ... existing code ...
    Some(entry) => match entry.value.as_redis_value() {
        RedisValueRef::Hash(map) => Ok(Some(map)),
        RedisValueRef::HashListpack(_) => Ok(None),  // treat empty/compact as not-yet-created
        _ => Err(Self::wrongtype_error()),
    }
}
```

Wait -- that loses data. Better approach: the _if_alive methods are used in the sharded read path. Since compact encodings will be rare in read-heavy paths (they upgrade on first write), and the sharded architecture uses mutable accessors for most operations, we can simply not support compact variants in _if_alive and treat them as needing mutable upgrade. For the _if_alive read path, if we encounter a compact variant, return None (the caller will fall through to the mutable path).

Actually, the cleanest solution: make the _if_alive methods handle compact variants by also returning a reference. For HashListpack, we can't return `&HashMap`. So we need to either:
1. Always upgrade on create (don't use compact for collections that might be read via _if_alive)
2. Add compact variants to the return type

Let's go with approach: for the read-only _if_alive methods, match on compact variants and return `Ok(None)`. The caller (sharded read path) will then fall through to the mutable path which upgrades. This is safe because newly-created compact collections have just been written via the mutable path and won't be hit by the read-only path until after upgrade.
