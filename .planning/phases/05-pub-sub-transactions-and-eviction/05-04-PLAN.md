2. **Pub/Sub integration tests:**

   `test_pubsub_subscribe_and_publish`:
   - Start server. Create two redis connections (use `redis::Client::get_multiplexed_async_connection` for the publisher, and `redis::Client::get_async_pubsub()` for the subscriber).
   - Subscriber: `pubsub.subscribe("test-channel").await`. Publisher: `redis::cmd("PUBLISH").arg("test-channel").arg("hello").query_async::<i64>(&mut pub_conn).await` -- assert returns 1 (one subscriber).
   - Subscriber: `pubsub.on_message().next().await` -- verify channel == "test-channel" and payload == "hello".
   - Use tokio::time::timeout to avoid hanging if message never arrives (e.g., 2 seconds).

   `test_pubsub_psubscribe`:
   - Start server. Two connections: subscriber uses `pubsub.psubscribe("news.*").await`.
   - Publisher publishes to "news.sports". Subscriber receives pmessage matching "news.*" with channel "news.sports".
   - Verify pattern and channel in received message.

   `test_pubsub_unsubscribe`:
   - Start server. Subscribe to "ch1". Unsubscribe from "ch1". Publish to "ch1" -- returns 0 (no subscribers).

   `test_pubsub_multiple_channels`:
   - Subscribe to "ch1" and "ch2". Publish to "ch1" -- subscriber gets message. Publish to "ch2" -- subscriber gets message. Both return 1.

   `test_pubsub_publish_returns_count`:
   - Two subscribers on same channel. PUBLISH returns 2.

3. **Transaction integration tests:**

   `test_multi_exec_basic`:
   - Start server. Open one connection (non-multiplexed: `redis::Client::get_async_connection().await`).
   - Send MULTI (OK), SET key1 val1 (QUEUED), SET key2 val2 (QUEUED), GET key1 (QUEUED), EXEC (returns array [OK, OK, "val1"]). Use `redis::cmd("MULTI")`, etc.
   - Verify results array matches expected.

   `test_multi_discard`:
   - MULTI, SET key1 val1 (QUEUED), DISCARD (OK). GET key1 returns nil (key was not set).

   `test_exec_without_multi`:
   - EXEC without MULTI returns error.

   `test_discard_without_multi`:
   - DISCARD without MULTI returns error.

   `test_nested_multi`:
   - MULTI, then MULTI again -- returns ERR.

   `test_watch_success`:
   - Start server. Use non-multiplexed connection. WATCH key1, MULTI, SET key1 "new", EXEC. No interference -- EXEC succeeds, returns array [OK]. GET key1 returns "new".

   `test_watch_abort`:
   - Start server. Two non-multiplexed connections (conn_a, conn_b).
   - conn_a: SET key1 "initial". conn_a: WATCH key1. conn_a: MULTI.
   - conn_b: SET key1 "modified" (interferes).
   - conn_a: SET key1 "from_a" (QUEUED). conn_a: EXEC -- returns Null (transaction aborted).
   - Verify: GET key1 returns "modified" (conn_b's value persisted, conn_a's transaction was discarded).

4. **Use non-multiplexed connections for transaction tests** because multiplexed connections share state and cannot track per-connection transaction state. Pattern:
   ```rust
   let client = redis::Client::open(format!("redis://127.0.0.1:{}/", port)).unwrap();
   let mut conn = client.get_async_connection().await.unwrap();
   ```
   Note: `get_async_connection` might be deprecated in redis 0.27 in favor of `get_multiplexed_async_connection`. Check if `get_tokio_connection` or similar exists. If only multiplexed available, use raw `redis::cmd` with pipeline or use a `redis::aio::Connection` directly. Alternatively, use `redis::Client::get_connection()` (sync) in a `tokio::task::spawn_blocking`.

   ```rust
   async fn start_server_with_maxmemory(maxmemory: usize, policy: &str) -> (u16, CancellationToken) {
       let listener = TcpListener::bind("127.0.0.1:0").await.unwrap();
       let port = listener.local_addr().unwrap().port();
       drop(listener);
       let token = CancellationToken::new();
       let server_token = token.clone();
       let config = ServerConfig {
           bind: "127.0.0.1".to_string(), port, databases: 16,
           requirepass: None,
           appendonly: "no".to_string(), appendfsync: "everysec".to_string(),
           save: None, dir: ".".to_string(), dbfilename: "dump.rdb".to_string(),
           appendfilename: "appendonly.aof".to_string(),
           maxmemory, maxmemory_policy: policy.to_string(), maxmemory_samples: 5,
       };
       tokio::spawn(async move { listener::run_with_shutdown(config, server_token).await.unwrap(); });
       tokio::time::sleep(std::time::Duration::from_millis(50)).await;
       (port, token)
   }
   ```

2. **CONFIG integration tests:**

   `test_config_get_maxmemory`:
   - Start server with default config (maxmemory 0). CONFIG GET maxmemory -- returns ["maxmemory", "0"].
   - Use `redis::cmd("CONFIG").arg("GET").arg("maxmemory").query_async::<Vec<String>>(&mut conn)`.

   `test_config_set_maxmemory`:
   - CONFIG SET maxmemory 1048576 (1MB). CONFIG GET maxmemory returns ["maxmemory", "1048576"].

   `test_config_get_glob_pattern`:
   - CONFIG GET "max*" -- returns all max-prefixed params (maxmemory, maxmemory-policy, maxmemory-samples).

   `test_config_set_maxmemory_policy`:
   - CONFIG SET maxmemory-policy allkeys-lru. CONFIG GET maxmemory-policy returns ["maxmemory-policy", "allkeys-lru"].

3. **Eviction integration tests:**

   `test_eviction_noeviction_rejects_writes`:
   - Start server with maxmemory=1024, policy="noeviction".
   - SET a key with a large value (e.g., 2000 bytes of "x") -- should succeed (first write, under limit after initial overhead clarification... actually the Entry overhead is ~128+key_len+val_len. If maxmemory=1024 and value=2000 bytes, it exceeds immediately).
   - Adjust: set maxmemory to a reasonable threshold. SET key1 with 500-byte value (succeeds, est ~630 bytes). SET key2 with 500-byte value -- should fail with OOM error if total > 1024.
   - Actually, use a simpler approach: set maxmemory=500, set a key with 200-byte value (succeeds ~330 bytes), then set another key (should fail with OOM since total > 500).
   - Verify OOM error message.

   `test_eviction_allkeys_lru`:
   - Start server with maxmemory=2048, policy="allkeys-lru".
   - SET key1 val1, SET key2 val2, SET key3 val3 (all should fit).
   - SET key_big with a large value that pushes over maxmemory -- should succeed because LRU evicts an older key.
   - Verify: at least one of key1/key2/key3 is gone (EXISTS returns 0).
   - OR simpler: use CONFIG SET to set a very small maxmemory after filling data, then do one more write and check eviction happened.

   `test_eviction_config_set_triggers_eviction`:
   - Start server with no maxmemory (default 0). SET key1..key10 with values. CONFIG SET maxmemory to a value smaller than current usage. Next write triggers eviction. Verify some keys evicted.

NOTE: Eviction tests are inherently probabilistic (random sampling). Use assertions that are robust: "at least one key was evicted" rather than "key1 was specifically evicted". Use a pattern like: count = number of keys before write, write big key, count after <= count before (some eviction happened).
