<objective>
Replace blocking file reads in replication full resync with async I/O and add optional Redis RDB format conversion for snapshot transfer.

Purpose: The full resync path in master.rs reads per-shard snapshot files from disk with blocking I/O (std::fs::read), which stalls the event loop during large transfers. Switching to async reads prevents this. Additionally, adding Redis RDB conversion enables interop with standard Redis replicas.

Output: Updated master.rs with async file reads and redis_rdb integration.
</objective>

**For the Tokio variant:**

1. Find the file read pattern. It likely looks like:
```rust
let data = std::fs::read(&shard_path)?;
```

Replace with:
```rust
let data = tokio::fs::read(&shard_path).await
    .context(format!("Failed to read snapshot file: {}", shard_path.display()))?;
```

2. Do this for ALL file reads in the full resync path. There should be one per shard (looping over shard_id 0..num_shards).

3. Add an optional Redis RDB conversion path. After reading all shard RRDSHARD files, if a flag or negotiation indicates the replica wants Redis RDB format, convert:

// Optional: convert per-shard RRDSHARD data to Redis RDB for standard Redis replicas

// The redis_rdb module is available for future use:
// use crate::persistence::redis_rdb;
```

Add a TODO comment for future streaming RDB conversion, but do NOT change the transfer format yet. The primary goal is async I/O.

**For the Monoio variant:**

1. Monoio doesn't have tokio::fs equivalent readily available. Keep `std::fs::read()` but add a comment explaining:

// Monoio: synchronous file read. Thread-per-core model means this

// monoio::fs::File with read_at() in the future.
```

2. Ensure the Monoio variant compiles and has the same logic structure as Tokio.
