  <action>
Create profile.sh as an executable bash script following bench-production.sh patterns (ports, cleanup trap, wait_for_server, get_rss_kb, format_number, ratio helpers).

**Configuration:**
- PORT_REDIS=6399, PORT_RUST_TOKIO=6400, PORT_RUST_MONOIO=6401
- RUST_BINARY=./target/release/rust-redis
- OUTPUT_FILE=PROFILING-REPORT.md
- SIZES=(10000 50000 100000 500000 1000000) for dataset scaling

**Script sections:**

1. **Header and argument parsing** — `--output FILE`, `--sizes "10000 50000"`, `--skip-monoio` (in case monoio build fails), `--section NAME` to run one section only.

2. **Server lifecycle helpers** — Reuse bench-production.sh patterns: start_redis(), start_rust_tokio(), start_rust_monoio(), cleanup(), wait_for_server(), flush_server().
   - start_rust_tokio: `$RUST_BINARY --port $PORT_RUST_TOKIO --shards 1 &`
   - start_rust_monoio: `$RUST_BINARY --port $PORT_RUST_MONOIO --shards 1 --runtime monoio &` (if binary supports it)
   - Build both: `cargo build --release --features runtime-tokio` and `cargo build --release --features runtime-monoio` (copy to separate binaries)

3. **Measurement helpers:**
   - `get_memory_info(port)` — parse `redis-cli -p $port INFO memory` for used_memory, used_memory_rss, mem_fragmentation_ratio
   - `get_cpu_pct(pid)` — `ps -o %cpu= -p $pid`
   - `measure_rss_delta(port, key_count, key_type)` — flush, record baseline RSS, insert N keys via redis-benchmark or redis-cli pipeline, record final RSS, compute delta per key

4. **Section: Per-Key Memory Overhead** (core measurement)
   For each size in SIZES and each type in (string, hash, list, zset):
   - Flush server
   - Measure baseline RSS via ps
   - Insert keys using redis-benchmark for strings, redis-cli EVAL pipeline for hashes/lists/zsets:
     - Strings: `redis-benchmark -p $port -t set -n $count -r $count -d 256 -q`
     - Hashes: `redis-cli -p $port --pipe` with HSET key field1..field5 val (5 fields, 256B total)
     - Lists: `redis-cli -p $port --pipe` with LPUSH key elem1..elem10 (10 elements)
     - Sorted sets: `redis-cli -p $port --pipe` with ZADD key score1 mem1..score10 mem10
   - Measure final RSS
   - Compute: per_key_bytes = (final_rss - baseline_rss) * 1024 / key_count
   - Also capture `redis-cli INFO memory` used_memory for comparison
   - Record results in arrays for table generation

5. **Section: Memory Fragmentation**
   - Insert 1M string keys with 256B values
   - Record RSS and used_memory
   - Delete 500K random keys (use redis-cli DEL in batches)
   - Record RSS and used_memory after deletion
   - Compute fragmentation_ratio = rss / used_memory
   - Compare rust-redis vs Redis

6. **Section: CPU Efficiency**
   For each server (Redis, Tokio, Monoio):
   - Start server, wait for ready
   - Record CPU% baseline via `ps -o %cpu=`
   - Run redis-benchmark mixed workload for 10 seconds: `-c 50 -n 200000 -t set,get -r 100000 -d 256`
   - Sample CPU% every second during benchmark (background sampler)
   - Record ops/sec from redis-benchmark output
   - Compute: efficiency = ops_per_sec / avg_cpu_pct

7. **Section: Existing Criterion Benchmarks**
   - Run `cargo bench --bench compact_key 2>&1` and capture output
   - Run `cargo bench --bench entry_memory 2>&1` and capture output
   - Run `cargo bench --bench get_hotpath 2>&1` and capture output
   - Parse Criterion output lines for the report

8. **Report generation** — Write PROFILING-REPORT.md with:
   - Header with date, hardware info (sysctl hw.ncpu, sysctl hw.memsize, sw_vers)
   - Per-Key Memory Overhead table (columns: Key Count, Type, Redis bytes/key, rust-redis bytes/key, Ratio)
   - RSS Growth Curve table (columns: Key Count, Redis RSS MB, rust-redis RSS MB)
   - Memory Fragmentation table (columns: Metric, Redis, rust-redis)
   - CPU Efficiency table (columns: Server, ops/sec, avg CPU%, ops/sec/CPU%)
   - Criterion Benchmarks section (paste parsed output)
   - Optimization Recommendations section (auto-generated based on ratios)

Make the script executable: chmod +x profile.sh.

IMPORTANT: Use `set -euo pipefail` at top. Use trap cleanup EXIT. All temporary files go in a RESULTS_DIR that gets cleaned up.
