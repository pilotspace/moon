**C) At the end of batch processing, after all responses have been sent via `framed.send()`/`framed.feed()`, ensure the BumpVec is dropped before the await and add arena reset.**

The responses Vec is consumed when sending. After the response sending section (look for the code that iterates responses and calls `framed.feed()` or `framed.send()`), add explicit drop and reset:

```rust
drop(responses); // Drop BumpVec before await (Send safety)
```

Then after the `framed.flush().await` (or equivalent send await), add:
```rust
arena.reset(); // O(1) bulk deallocation of batch temporaries
```

**CRITICAL Send safety rule:** The `BumpVec<Frame>` (which is `!Send`) MUST be dropped before any `.await` point. The `responses` variable must not be held across an await. If the current code pattern sends responses in a loop with awaits, restructure so that:
1. All responses are collected in the BumpVec (sync)
2. Responses are copied/moved to the framed sink (the `framed.feed()` loop is sync since feed doesn't await)
3. BumpVec is dropped
4. `framed.flush().await` happens AFTER drop

The non-sharded handler already follows this pattern correctly (see lines 618-715). Mirror that approach.

**IMPORTANT:** bumpalo::Bump and BumpVec are already imported at the top of connection.rs (lines 1-2). No new imports needed.
