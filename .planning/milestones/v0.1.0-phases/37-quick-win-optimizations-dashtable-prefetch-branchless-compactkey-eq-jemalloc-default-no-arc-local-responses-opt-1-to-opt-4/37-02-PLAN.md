  <name>Task 2: Remove unnecessary frame clone in monoio cross-shard dispatch (OPT-4)</name>
  <files>src/server/connection.rs</files>
  <read_first>src/server/connection.rs (lines 3639-3650)</read_first>
  <action>
In `handle_connection_sharded_monoio`, line 3644 has:
```rust
command: std::sync::Arc::new(frame.clone()),
```

The `frame` is already owned (consumed from the batch iteration). The `.clone()` performs an unnecessary deep copy of the entire Frame::Array (heap Vec + all inner Frames). Change to:
```rust
command: std::sync::Arc::new(frame),
```

However, verify that `frame` is not used after this point in the same branch. If the compiler complains about a move, check whether `frame` is referenced later in the `else if` block. Based on the code at lines 3639-3680, the `frame` variable is only used at:
- Line 3644: `Arc::new(frame.clone())` -- the clone
- Line 3673: `aof::serialize_command(frame)` -- this also needs the frame

Since `serialize_command` takes `&Frame`, it should work with a reference. Check the signature. If `serialize_command` takes `&Frame`, then:
1. Move `frame` into the Arc: `let arc_cmd = std::sync::Arc::new(frame);`
2. Use `&*arc_cmd` for serialize_command: `aof::serialize_command(&*arc_cmd)`
3. Pass `arc_cmd` into the ShardMessage

If this gets complicated or the borrow checker fights it, the simpler approach: keep `frame.clone()` but note in the SUMMARY that this optimization is already effectively a no-op for local commands (95%+ path) since local dispatch in both tokio and monoio handlers uses plain `Frame` without any Arc. The Arc overhead exists only on the cross-shard path which is already the minority case.

NOTE: The OPTIMIZATION-ANALYSIS.md describes OPT-4 as "every command response wrapped in Arc<Frame>" but code review shows this is NOT the case in the current codebase. Local commands already return plain `Frame`. Arc is only used for cross-shard command dispatch (not responses). The optimization is already effectively implemented for the local path.
