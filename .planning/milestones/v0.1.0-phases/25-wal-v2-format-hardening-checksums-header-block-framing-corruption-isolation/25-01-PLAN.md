Modify `src/persistence/wal.rs` to implement WAL v2 format:

**1. Add constants at module top:**
```rust
/// WAL v2 format constants
const WAL_MAGIC: &[u8; 6] = b"RRDWAL";
const WAL_VERSION: u8 = 2;
const WAL_HEADER_SIZE: usize = 32;
/// Block overhead: block_len(4) + cmd_count(2) + db_idx(1) + crc32(4) = 11 bytes
const BLOCK_OVERHEAD: usize = 11;
```

**2. Add `cmd_count: u16` field to `WalWriter` struct** to track commands appended since last flush. Initialize to 0 in `new()`.

**3. Add `header_written: bool` field to `WalWriter` struct** to track whether the v2 header has been written to the current file. Initialize based on `write_offset == 0`.

**4. Add private `write_header(&mut self) -> io::Result<()>` method:**
- Writes 32 bytes: `WAL_MAGIC`(6B) + `WAL_VERSION`(1B) + `shard_id as u16` in LE(2B) + `epoch` as u64 LE(8B) + 13 zero bytes (reserved) + 2 zero bytes padding to reach 32.
- Wait: 6+1+2+8+13 = 30. Add 2 more reserved bytes to reach 32.
- Actually: 6+1+2+8 = 17, so reserved = 15 bytes to reach 32.
- Writes directly to file (not through buf), updates write_offset.
- Sets `header_written = true`.

**5. Modify `new()` constructor:**
- After opening the file, if `write_offset == 0` (fresh file), call `write_header()`.
- If `write_offset > 0` (existing file being appended to), set `header_written = true` (assume existing file already has header or is v1 -- replay handles detection).

**6. Modify `append(&mut self, data: &[u8])`:**
- Keep existing `self.buf.extend_from_slice(data)`.
- Add `self.cmd_count += 1` (or saturating_add to avoid overflow).

**7. Modify `do_write(&mut self) -> io::Result<()>`:**
- Before writing, if `!self.header_written`, call `self.write_header()?`.
- Instead of writing raw `self.buf`, construct a block frame:
  - Let `payload = &self.buf` (the accumulated RESP bytes).
  - Compute CRC32 over: cmd_count(2B LE) + db_idx(1B, always 0) + payload.
    Use `crc32fast::hash()` -- build a hasher, feed cmd_count bytes, feed db_idx byte, feed payload, finalize.
  - Compute `block_len = 2 (cmd_count) + 1 (db_idx) + payload.len() + 4 (crc32)` as u32 LE.
  - Write to file in order: block_len(4B LE), cmd_count(2B LE), db_idx(1B = 0x00), payload, crc32(4B LE).
  - Use a small stack buffer for the 7-byte header (block_len + cmd_count + db_idx), then write payload, then write crc32. Or use `write_all` three times. Or build a single Vec -- but avoid allocation on the hot path. Best approach: use `std::io::Write` with three write_all calls (OS will coalesce in page cache).
- Update `write_offset` by `4 + block_len` (the 4-byte block_len prefix is NOT included in block_len value, OR it IS -- need to decide). Design says `block_len covers: cmd_count + db_idx + RESP payload + CRC32`. So block_len = 2+1+payload.len()+4. The on-disk layout is: [block_len:4][cmd_count:2][db_idx:1][payload:var][crc32:4]. Total bytes = 4 + block_len.
- Update `bytes_written` by total bytes written (4 + block_len).
- Reset `self.cmd_count = 0`.
- Clear `self.buf` (keep allocation).

**8. Modify `truncate_after_snapshot()`:**
- After opening the fresh WAL file, set `header_written = false`.
- The next `do_write()` or explicit call will write the header.
- Actually, write header immediately: call `self.write_header()?` after opening the new file.

**9. Add `use crc32fast;` import** (already in Cargo.toml deps).

**10. Update existing unit tests** to account for the new binary format:
- `test_wal_append_and_flush`: The file contents will now have header + block frame, not raw RESP. Update assertion to check file starts with b"RRDWAL" and contains the RESP payload within a block.
- `test_wal_multiple_appends_batched`: Update to verify file starts with header, then a single block containing all 100 commands.
- `test_wal_flush_empty_is_noop`: Should still work (no block written for empty buf).
- `test_wal_truncate_after_snapshot`: Update to verify new file has header (32 bytes), old file has header + data.
- `test_wal_path_construction`: No change needed.
- `test_wal_replay_round_trip` and `test_wal_replay_with_collections`: These call `replay_wal()` which still uses `replay_aof()` -- they will BREAK because replay doesn't understand v2 format yet. **Temporarily skip or remove these replay tests** -- Plan 02 will implement v2 replay and restore+update them. Mark with `#[ignore]` and a comment: `// TODO: Plan 02 implements v2 replay`.

**Add new unit test `test_wal_v2_header_format`:**
- Create writer, flush one command, read file bytes.
- Assert first 6 bytes == b"RRDWAL".
- Assert byte 6 == 2 (version).
- Assert bytes 7..9 == shard_id as u16 LE.
- Assert bytes 9..17 == epoch as u64 LE.
- Assert bytes 17..32 are all zero (reserved).

**Add new unit test `test_wal_v2_block_crc_valid`:**

- Read file, skip 32-byte header.
- Parse block_len (4B LE), cmd_count (2B LE), db_idx (1B), extract payload, read crc32 (4B LE).
- Recompute CRC over cmd_count+db_idx+payload, assert matches.
