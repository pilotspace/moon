<objective>
Port pipeline batch dispatch from Tokio handler to Monoio handler in connection.rs. The Monoio handler currently dispatches each remote command individually as ShardMessage::Execute with synchronous await inside the frame loop, causing negative scaling under multi-shard workloads. The Tokio handler already implements the correct pattern: defer remote commands into a HashMap<usize, Vec<...>> grouped by target shard, then send one PipelineBatch per shard and await all responses in parallel.

Also fix the `for frame in &frames` borrow pattern that forces `frame.clone()` on every remote dispatch -- change to `for frame in frames` to consume the iterator.

Purpose: Eliminate the #1 performance bottleneck causing negative shard scaling in the Monoio runtime path.
Output: Updated handle_connection_sharded_monoio with batched pipeline dispatch matching Tokio handler pattern.
</objective>
