2. Create `src/runtime/monoio_impl.rs` mirroring tokio_impl.rs structure:

   - `MonoioTimer` struct implementing `RuntimeTimer`:
     - `interval(period)` -> wraps `monoio::time::interval(period)` in `MonoioInterval`
     - `sleep(duration)` -> `Box::pin(monoio::time::sleep(duration))`

   - `MonoioInterval` struct wrapping `monoio::time::Interval`:
     - `tick()` -> `Box::pin(async { self.0.tick().await; })`

   - `MonoioSpawner` struct implementing `RuntimeSpawn`:
     - `spawn_local(future)` -> `monoio::spawn(future);` (monoio::spawn is inherently local)

   - `MonoioRuntimeFactory` struct implementing `RuntimeFactory`:
     - `block_on_local(name, f)` -> build monoio runtime via `monoio::RuntimeBuilder::<monoio::FusionDriver>::new().enable_timer().build().expect(...)` then call `runtime.block_on(f)`. Note: monoio RuntimeBuilder does not have a thread_name method, so `name` parameter is used only in the panic message. Use `FusionDriver` for io_uring on Linux + kqueue on macOS.

   - `MonoioFileIo` struct implementing `FileIo`:
     - `type Writer = std::fs::File;` (same as Tokio -- sync file ops)
     - `open_append(path)` -> `std::fs::OpenOptions::new().create(true).append(true).open(path)`
     - `open_write(path)` -> `std::fs::OpenOptions::new().create(true).write(true).truncate(true).open(path)`

   - `FileSync for std::fs::File` -- this is ALREADY implemented in tokio_impl.rs. Use a conditional compilation approach: only implement in monoio_impl.rs if runtime-tokio is NOT active, to avoid duplicate impl. Use:
     ```rust
     #[cfg(not(feature = "runtime-tokio"))]
     impl FileSync for std::fs::File {
         fn sync_data(&self) -> io::Result<()> {
             std::fs::File::sync_data(self)
         }
     }
     ```
     This ensures exactly one impl exists regardless of which feature is active.
