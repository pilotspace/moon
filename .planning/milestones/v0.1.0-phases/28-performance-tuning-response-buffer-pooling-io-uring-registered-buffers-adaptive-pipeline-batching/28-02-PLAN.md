<objective>
Add adaptive pipeline batching to the io_uring command dispatch path and optimize response buffer reuse in the Tokio connection handler.

Purpose: The io_uring path currently dispatches commands one-at-a-time inside the parse loop, acquiring borrow_mut on databases for each command. Batching all parsed frames first, then dispatching under a single borrow_mut, reduces RefCell overhead and improves cache locality for pipelined workloads. The Tokio path already batches but can benefit from write_buf capacity retention.

Output: io_uring handle_uring_event batches parsed commands before dispatch; Tokio handler retains write_buf capacity across iterations.
</objective>

Refactor `handle_uring_event` IoEvent::Recv handler to batch-dispatch commands:

1. Currently the code parses and dispatches in a tight loop (lines 988-1069): parse frame -> borrow_mut databases -> dispatch -> serialize -> submit_send. Each iteration acquires and releases the RefCell borrow.

2. Change to a two-phase approach:
   Phase A — Parse all frames into a `Vec<Frame>` (up to 1024, matching MAX_BATCH from the Tokio path):
   ```
   let mut batch: Vec<Frame> = Vec::with_capacity(16);
   loop {
       match parse(parse_buf, &parse_config) {
           Ok(Some(frame)) => { batch.push(frame); if batch.len() >= 1024 { break; } }
           Ok(None) | Err(ParseError::Incomplete) => break,
           Err(_) => { /* close connection */ return; }
       }
   }
   if batch.is_empty() { return; }
   ```

   Phase B — Dispatch all commands under a single borrow_mut:
   ```
   let responses: Vec<Frame> = {
       let mut dbs = databases.borrow_mut();
       let db_count = dbs.len();
       dbs[0].refresh_now();
       let mut selected = 0usize;
       batch.iter().map(|frame| {
           let (cmd, args) = match Self::extract_command_static(frame) {
               Some(pair) => pair,
               None => return Frame::Error(Bytes::from_static(b"ERR invalid command")),
           };
           match cmd_dispatch(&mut dbs[0], cmd, args, &mut selected, db_count) {
               DispatchResult::Response(f) => f,
               DispatchResult::Quit(f) => f,
           }
       }).collect()
   };
   ```

   Phase C — Serialize and submit all responses (outside borrow):
   Iterate responses, use existing BulkString writev path or submit_send path per response.

3. This matches the pattern already used in `handle_connection_sharded` (the Tokio path) which parses all frames into `batch`, then dispatches under a single lock.

4. For the io_uring send path: when Plan 01 is also applied, use pooled buffers. But since this plan has no dependency on Plan 01, use the existing BytesMut allocation path. The two plans compose cleanly — Plan 01 changes HOW buffers are allocated, this plan changes WHEN dispatch happens.

5. Do NOT change the BulkString writev scatter-gather path — that remains per-response since each writev needs its own iovec array.

2. Add capacity-shrink guard: after write_all, if `write_buf.capacity() > 65536`, shrink it back to 8192 to prevent unbounded growth from large pipeline responses:
   ```rust
   if write_buf.capacity() > 65536 {
       write_buf = BytesMut::with_capacity(8192);
   }
   ```
   Place this after the `write_all` call at line 2325.

3. Apply the same shrink guard to the non-sharded `handle_connection` path if it exists with similar write_buf pattern.

4. For the `read_buf`, apply the same shrink guard (if capacity > 65536, reallocate to 8192) after processing a batch, to prevent clients that send one huge pipeline from permanently inflating the read buffer.

5. This is a small but important optimization for long-lived connections that occasionally see large pipelines — prevents permanent memory bloat per connection.
